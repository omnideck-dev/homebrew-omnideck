### 2. Homebrew Core Rules

* **Formula vs. Cask:** The tool distributed here (`omnideck`) is a CLI binary that interacts with containers. It is implemented as a standard **Formula**, *not* a Cask. Do not convert this repository or formula into a Cask structure.
* **Naming Conventions:** - GitHub Repository: `homebrew-omnideck` (strips down to the namespace `omnideck-dev/omnideck` when users invoke `brew install`).
* Ruby Class Name: Must match the file name in strict CamelCase. For `Formula/omnideck.rb`, the class definition must be `class Omnideck < Formula`.



### 3. Formula Automation Guidelines

When updating the `Formula/omnideck.rb` file for new releases:

* **Source Artifacts:** Use the stable tarballs generated by GitHub releases from `omnideck-dev/cli` (e.g., `https://github.com/omnideck-dev/cli/archive/refs/tags/vX.Y.Z.tar.gz`).
* **Cryptographic Hashes:** Every version bump requires updating the `sha256` string. Never commit a formula with an unverified or placeholder SHA-256 hash.
* **Dependencies:** The CLI tool relies on `go` for compilation if built from source. Ensure `depends_on "go" => :build` is explicitly retained unless shifting exclusively to pre-compiled multi-architecture binary distribution.

## Common Workflows & Routine Tasks

### Bumping the CLI Version

When tasked with updating the formula for a new release of the Omnideck CLI, execute the following logic:

1. Fetch the target version tag (e.g., `v1.0.1`).
2. Construct the source archive URL: `https://github.com/omnideck-dev/cli/archive/refs/tags/v1.0.1.tar.gz`.
3. Compute the SHA-256 hash of that archive.
4. Modify `Formula/omnideck.rb` by updating the `url` and `sha256` fields.
5. Verify the formula syntax locally if a Homebrew environment is active: `brew audit --new-formula Formula/omnideck.rb`.
